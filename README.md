# hadith-db-v01

A complete **verse → hadith reference** for the ped-git hadith viewer (`hadith.html`,
"بازسازی شده" source). For every Qur'an verse it lists *every* hadith in the corpus that
quotes that verse, each rendered as a consensus set: text-reuse clusters as multi-member
sets, standalone hadiths ("singletons") as size-1 sets. Per-segment Persian translation is
included where available; Arabic-only sets are normal.

Generated by `src/export_csquranics.py` in the hadith-hub project from the consensus
clusters (passim recall → refine → exact-dup/isnad → alignment-fold) plus all verse-quoting
singletons. Cross-membership is partitioned (each hadith → its largest cluster) so every set
has a unique `translation_source_hid` and each verse lists each hadith once.

## Layout

```
consensus/aye_hadith_ref.json   { "<sura:aya>": { "hadith_ids": [hid, ...] } }
consensus/h/<setId>.json        { "body_consensus_columns": [ <column> ],
                                  "members": [ {header_text, body_text, translation, aligned_body} ] }
translate/translate.json        { "items": [ {consensus_set_id, detail_file, member_count,
                                  segment_count, translation_available_count, translation_source_hid} ] }
translate/d/<setId>.json        { consensus_set_id, translation_language, translation_source{hid},
                                  segments:[{source_hadith_segment, translation_segment,
                                             segment_columns:[ <column> ]}] }
```

A set is shown under an ayah when its `translation_source_hid` is listed in
`aye_hadith_ref["<sura:aya>"].hadith_ids`. Set ids: clusters use the numeric cluster id;
singletons use `s<hadith_id>`. Files are sharded one-per-set so no single file is large.

## Alignment grid

So a set's members can be rendered as column-aligned views (per-segment alignment, word/token
alignment, aligned hadith list), every member is laid onto **one shared set of `C` columns** by a
multiple-sequence alignment of the matn. `body_text` stays the raw matn (with inline
`<quranic_text>…</quranic_text>` and `<footnote>[…]</footnote>`) for the plain list; the grid is added
on top:

- **`body_consensus_columns`** — `C` column descriptors for the merged/consensus row.
- **`segment_columns`** (on each `translate/d` segment) — the display member's segments partition the
  columns; concatenating `segment_columns` across all segments, in order, equals `body_consensus_columns`.
- **`aligned_body`** (per member) — the same `C` columns as a whitespace-separated string of cells, each
  cell one of: a **word** (the member's token in that column), **`-`** (a gap — this member has nothing
  there), or **`TAG_OPEN:<tag>` / `TAG_CLOSE:<tag>`** (a markup boundary), `tag ∈ {quranic_text, footnote}`.

A **column descriptor** is one object:

```
{ "kind": "word", "token": "<normalized>", "display": "<text to show>" }   // show display || token
{ "kind": "tag_open" | "tag_close", "tag_name": "quranic_text" | "footnote" }  // emit <tag> / </tag>
```

Invariants: `C = len(body_consensus_columns) = Σ len(segment.segment_columns) = ` the cell count of
every member's `aligned_body`; column `j` means the same thing in the consensus row and in every
member (its word, `-`, or the same `TAG_*`); tag columns are matched open/close pairs. The grid may be
absent on a few sets (e.g. trivial singletons or unalignable markup) — render the plain list from
`body_text` when `body_consensus_columns` is missing.

Segment text is normalised so each verse reads cleanly:
- a verse / footnote the segmenter split across a chunk boundary is reunited, so the `»`, the
  ayah number, and the `[surah ayah]` footnote always sit on the verse's own chunk;
- **consecutive Qur'anic spans that resolve to the same ayah are coalesced into one quote with a
  single footnote** (e.g. `«وَ مِنْ»[المؤمنون۱۰۰] «وَرٰائِهِمْ … يُبْعَثُونَ»[المؤمنون۱۰۰]`
  → `«وَ مِنْ وَرٰائِهِمْ … يُبْعَثُونَ»[المؤمنون۱۰۰]`);
- where the upstream segmentation is unfaithful (drops/mangles markup so a span cannot be closed),
  the chunk display falls back to the balanced full body, so no broken markup is ever emitted.

Every `<quranic_text>` and `<footnote>` is balanced in every chunk across all sets.
